# 课题一产品与技术总体设计

更新时间：2026-06-19  
系统名称：面向园区网络的全流量采集分析系统  
适用课题：课题一“园区网络流量智能检测与分析”  
文档定位：合并后的 Markdown 主线，用于承接产品设计、技术设计、工程闭环、评审整改和验收证据包。

## 1. 设计结论

本系统应被定义为“课题一验收导向的园区全流量安全运营闭环系统”，而不是单纯的流量可视化平台、抓包平台、IDS 或 SIEM。

核心价值链：

```text
全流量可见
  -> 多源融合
  -> 智能检测
  -> 告警研判
  -> PCAP / 图谱取证
  -> 反馈学习
  -> 规则 / 模型治理
  -> 第三方可复测证据
```

当前工程已经具备较强的功能主链路证据。根据 `agent.md` 的 2026-06-19 状态记录，`tests/run_tests.sh full`、`make python-test`、`tests/run_tests.sh live` 均通过，live smoke 为 `7014/7014` checks passed。但这只能证明“功能主链路具备强证据”，不能证明“10 x 100Gbps、512Mpps、P95 <= 60s、95% 准确率、误报率 <5%、第三方测试、生产安全和 HA 验收已经达成”。

最终汇报措辞必须分层：

| 口径 | 可用措辞 | 禁止写法 |
|---|---|---|
| 功能主链路 | 已通过真实链路验证 | 已通过全部验收 |
| 100 轮 smoke | 具备功能回归强证据 | 性能/算法/安全指标已达成 |
| 性能/算法/安全/HA | 待专项验收 | 架构支持即实测达成 |
| 95%/5% 和第三方测试 | 待第三方验证 | 小样本 F1 证明任务书达标 |

## 4. 产品设计

### 4.1 产品定位

系统面向高教园区、科研园区、产业园区和政企园区，解决五类问题：

| 问题 | 产品能力 |
|---|---|
| 高带宽下看不清 | 全流量采集、会话化、流量态势、采集健康 |
| 告警多而不准 | 规则 + 行为模型 + CEP + 多源上下文 + 反馈学习 |
| 证据难回放 | Evidence、PCAP 索引/裁剪/下载、对象 hash、审计 |
| 关联分析依赖专家 | 攻击链、实体图谱、资产/用户/日志上下文 |
| 验收材料难闭环 | 指标追溯、证据包、试点材料、第三方测试口径 |

### 4.2 用户角色

| 角色 | 要完成的事 | 必须闭环 |
|---|---|---|
| 安全值班员 | 快速判断当前风险态势 | 大屏、仪表盘、采集健康、Top 风险 |
| 安全研判员 | 把告警判成 TP/FP/观察 | 告警详情、证据、PCAP、图谱、反馈 |
| 平台管理员 | 管理探针、规则、模型、白名单和部署 | 版本、灰度、回滚、审计 |
| 测试/验收人员 | 判断任务书指标是否成立 | 指标矩阵、样本、报告、证据包 |
| 园区负责人 | 理解投入价值和运营效果 | 报告、趋势、MTTR、误报率、试点效果 |

## 5. 技术设计

### 5.1 主链路

```text
Rust Probe Agent
  -> Ingest Gateway
  -> Kafka
  -> Flink Session / Feature / Rule / Behavior / CEP / Alert / Log / User Jobs
  -> ClickHouse / PostgreSQL / OpenSearch / NebulaGraph / Redis / MinIO
  -> Go control-plane APIs
  -> Web UI
  -> Feedback / Whitelist / Rule Review / MLOps
  -> Kafka rule.updates / model-updates
  -> Flink 热更新
```

设计原则：

| 原则 | 要求 |
|---|---|
| 契约优先 | `proto/traffic/v1` 是跨语言真源；REST、TS 类型、Kafka topic、DDL 应保持可追溯 |
| 流批结合 | 实时检测走 Kafka/Flink，离线回放和第三方测试走冻结样本和回放脚本 |
| 安全内建 | mTLS、JWT/OIDC、API Token、RBAC、tenant_id、Kafka TLS/SASL、ExternalSecret |
| 证据可追踪 | 每条告警追溯到 tenant、event、community、session、feature、rule/model、evidence |
| 可恢复 | DLQ、幂等键、checkpoint、savepoint、故障演练、RTO/RPO |

### 5.2 多源数据采集分工

| 数据源 | 采集/生产者 | Topic/接口 | 下游 | 产品呈现 |
|---|---|---|---|---|
| 流量数据 | Rust Probe + Ingest | `flow.events.v1` | Session、Feature、PCAP | 流量态势、告警、取证 |
| 资产信息 | Probe 被动发现、Asset Service、人工录入 | `asset.bindings.v1` / Asset API | PG、Graph、Alert Enrich | 资产台账、资产画像 |
| 设备日志 | Fluent Bit / Collector | `device.logs.v1` | Log Job、OpenSearch、CH | 日志上下文、暴破/外联辅助 |
| 用户行为 | Keycloak、APISIX、Auth Service | `user.events.v1` / Audit | User Behavior Job、Audit | 用户异常、权限变更、审计 |
| 反馈样本 | Alert Service / UI | `alert.feedback.v1` | MLOps、规则复审、指标 | TP/FP、原因码、白名单草案 |
| 模型/规则更新 | Rule Manager / MLOps | `rule.updates`、`model-updates` | Flink Rule/Behavior | 版本、灰度、回滚 |

### 5.3 数据存储

| 存储 | 主要数据 | 设计要求 |
|---|---|---|
| ClickHouse | flows、sessions、features、alerts、evidence、campaigns、pcap_index、dlq | 分布式表、批量写入、查询分区、schema diff |
| PostgreSQL | tenants、users、api_tokens、probes、assets、rules、models、deployments、audit | 元数据一致性、事务、备份/PITR |
| OpenSearch | 告警/证据/日志全文检索 | TLS/RBAC、模板、生命周期、降级策略 |
| NebulaGraph | 实体、关系、攻击路径 | schema 版本、慢查询保护、路径上限 |
| Redis | 缓存、去重、限流、会话 | key 规范、TTL、Sentinel/HA |
| MinIO | PCAP、模型、报告 | TLS、bucket policy、lifecycle、hash、签名 URL |

### 5.4 核心功能闭环

| 功能点 | 流程走向 | 闭环证据 |
|---|---|---|
| 采集到告警 | Probe -> Ingest -> Kafka -> Flink -> Alert -> UI | topic offset、Flink checkpoint、CH/OS 记录、UI 告警 |
| 告警到取证 | Alert -> Evidence -> PCAP Index -> Forensics -> MinIO -> Audit | PCAP hash、下载审计、session/community 对账 |
| 图谱关联 | Alert/Asset/Session -> NebulaGraph -> Graph API/UI | 节点/边/路径、层级限制、慢查询保护 |
| 反馈到优化 | TP/FP -> Feedback -> Whitelist/Rule Review/MLOps -> rule/model update | 反馈记录、白名单草案、模型评估、版本回滚 |
| 数据质量到恢复 | DLQ/迟到/解析失败 -> DataQuality -> 修复 -> replay -> 对账 | DLQ envelope、重放 offset、幂等和无重复报告 |
| 审计到合规 | 高风险操作 -> Audit -> Report -> Export | request_id、操作者、时间窗、资源、结果、report_id |

## 6. 验收设计

### 6.1 证据分层

| 层级 | 可证明内容 | 不能证明内容 |
|---|---|---|
| Smoke | 主链路未断、接口可用、页面可访问 | 性能、算法质量、生产安全、第三方验收 |
| Regression | 版本变更未破坏核心功能 | 10 x 100Gbps、95%/5% |
| Acceptance | 任务书专项指标可复测 | 第三方认证结论 |
| Third-party | 外部机构认可、试点材料、经济效益 | 代码质量和后续持续运行 |

### 6.3 检测质量门禁

检测质量不能只看单个 F1 值。任务书口径至少需要：

| 指标/证据 | 要求 |
|---|---|
| Accuracy | >= 95%，按冻结样本集和第三方口径计算 |
| FPR/误报率 | < 5%，明确 `FP/(FP+TN)` 或第三方约定口径 |
| Unknown Recall | 未知攻击开放集单独报告 |
| 样本量 | 正常、已知、未知、加密/隧道样本分别设最小样本量 |
| 置信区间 | 95% CI，样本不足自动失败 |
| 阈值 | 按 FPR <= 5% 约束选择，不以最大 F1 单独决定 |
| 离在线一致性 | Python/Java/Flink 同样本预测一致 |
| 回放可复现 | 冻结数据、代码、镜像、模型、阈值、规则、回放脚本 |

## 8. 竞争力设计

| 对比对象 | 常见弱点 | 本系统竞争点 | 证明材料 |
|---|---|---|---|
| SIEM | 偏日志汇聚，缺原始网络证据 | 全流量遥测 + PCAP 证据 + 图谱追踪 | Flow/Session/PCAP/Graph 证据链 |
| IDS | 单点规则命中，误报沉淀弱 | 规则 + CEP + 行为模型 + 反馈学习 | Rule/Behavior/CEP/MLOps |
| NDR | 场景宽泛，课题/试点材料弱 | 园区多源融合、任务书指标追溯、试点验收 | REQ-T1 矩阵、试点材料 |
| 抓包平台 | 查包依赖专家，运营闭环弱 | 从包到告警、证据、规则/模型优化 | Evidence、Forensics、Feedback |

建议统一对外表达：

> 本系统以全流量遥测为底座，以多源融合和实时流计算为核心，以告警证据链、PCAP 取证、实体图谱和反馈学习为闭环，面向园区网络形成“发现-研判-取证-优化-验收”的工程系统。

## 10. 文档合并与维护规则

本文件是当前 Markdown 主线。维护规则：

1. 新增产品、技术、验收、评审内容，优先更新本文件。
2. 正式 DOCX 从本文件和 PRD/SDD 模板同步生成，不再多处分叉手写。
3. `03_review/专家深评整改清单.md` 作为整改明细，不直接复制全量表格到主设计正文。
4. `02_acceptance/README.md` 作为证据包目录，不把证据文件散落在其他目录。
5. `04_assets/generated/` 和 `04_assets/diagrams/` 只存图形资产和图源，主文档通过相对路径引用。
6. 任何“已达成”表述必须能追溯到证据文件、测试报告或第三方报告。
