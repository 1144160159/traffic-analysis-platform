# 面向园区网络的全流量采集分析系统 UI 前端规范

更新时间：2026-06-26

视觉参考图：`../04_assets/generated/campus_full_traffic_system_visual_reference_20260620_business_corrected.png`

适用前端：`web/ui`，React + Ant Design + ECharts。

## 1. 设计目标

本规范用于把 GPT 视觉参考图转化为可持续实现的前端 UI 标准。目标不是只复刻一张大屏，而是建立统一的产品视觉语言，覆盖 `/screen`、`/dashboard`、`/alerts`、`/alerts/:alertId`、`/forensics`、`/graph`、`/assets`、`/data-quality`、`/compliance`、`/audit-log` 等业务页面。

核心体验：

- 一眼看清园区全流量采集链路是否健康。
- 一屏串联告警、资产、证据、响应和反馈学习。
- 所有页面共享同一套深色安全运营视觉 token。
- 前端组件密度高但不拥挤，适合值班、研判、审计和验收反复使用。

## 2. 信息架构

### 2.1 产品标题

产品标题固定为：

```text
园区网络全流量采集与分析系统
```

禁止使用“指挥研判平台”“Command Workbench”等作为页面主标题。Command Workbench 只作为内部设计方向，不进入产品导航。

### 2.2 一级菜单

一级菜单必须使用业务域：

| 菜单 | 对应路由 |
|---|---|
| 综合态势 | `/dashboard`、`/screen`、`/topics` |
| 采集监测 | `/probes`、`/data-quality` |
| 威胁分析 | `/alerts`、`/campaigns`、`/attack-chains`、`/encrypted-traffic`、`/forensics` |
| 资产图谱 | `/assets`、`/graph`、`/fusion`、`/baselines` |
| 检测运营 | `/rules`、`/deployments`、`/models`、`/mlops`、`/playbooks`、`/whitelist` |
| 审计配置 | `/compliance`、`/audit-log`、`/notifications`、`/settings` |

禁止把“看见、研判、取证、治理、验收”作为一级菜单。

### 2.3 二级菜单

二级菜单必须按一级业务域展开：

| 一级菜单 | 二级菜单 |
|---|---|
| 综合态势 | 仪表盘、态势大屏、专题面板 |
| 采集监测 | 探针管理、数据质量 |
| 威胁分析 | 告警中心、战役列表、攻击链分析、加密流量、取证分析 |
| 资产图谱 | 资产台账、实体图谱、数据融合、行为基准 |
| 检测运营 | 规则管理、部署管理、模型管理、MLOps 编排、SOAR 剧本、白名单 |
| 审计配置 | 合规审计、审计日志、通知配置、系统设置 |

当前页面高亮采用左侧蓝色激活条、浅蓝图标底、深蓝边框和计数角标组合。

## 3. 视觉 Token

### 3.1 色彩

| Token | 建议值 | 用途 |
|---|---:|---|
| `--ui-bg-page` | `#03111c` | 页面深色底 |
| `--ui-bg-shell` | `#061827` | 应用框架底 |
| `--ui-bg-panel` | `rgba(6, 28, 43, 0.86)` | 面板背景 |
| `--ui-bg-panel-strong` | `#071f32` | 重点面板和右侧详情 |
| `--ui-border-subtle` | `rgba(56, 151, 201, 0.22)` | 普通边框 |
| `--ui-border-active` | `#1e9cff` | 激活边框 |
| `--ui-glow-cyan` | `rgba(0, 191, 255, 0.45)` | 数据流与悬停光 |
| `--ui-text-primary` | `#eaf7ff` | 主文字 |
| `--ui-text-secondary` | `#9db9c9` | 次级文字 |
| `--ui-text-muted` | `#5e7b8d` | 弱信息 |
| `--ui-info` | `#18a8ff` | 信息/低危/链接 |
| `--ui-success` | `#36d66b` | 正常/通过/健康 |
| `--ui-warning` | `#ffb020` | 中危/注意 |
| `--ui-danger` | `#ff4d4f` | 高危/失败 |
| `--ui-critical` | `#ff2d2d` | 严重风险 |

约束：

- 背景以深青黑为主，不使用大面积紫色渐变、米色、棕橙色或营销式亮色背景。
- 高危红、中危黄、低危蓝、健康绿必须语义一致，不能随页面变化。
- 图表线条以蓝、青、绿为主，红色只用于风险和异常。

### 3.2 字体

建议字体栈：

```css
font-family: "Microsoft YaHei", "PingFang SC", "Noto Sans CJK SC", -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

字号层级：

| 层级 | 桌面字号 | 字重 | 用途 |
|---|---:|---:|---|
| 产品标题 | `24px` | 700 | 左上产品名 |
| 顶部 KPI 数字 | `20px` | 700 | 高风险、告警总数、健康度 |
| 面板标题 | `15px` | 600 | 所有 Panel 标题 |
| 表格正文 | `12px` | 400 | 表格、详情字段 |
| 标签/辅助 | `11px` | 400 | 图例、单位、说明 |
| 大屏指标 | `24-28px` | 700 | 圆环、核心指标 |

硬性要求：

- 所有面板标题统一为中文标题，不出现“中文 / English”形式。
- 面板标题视觉高度必须与视觉参考图中的“园区拓扑总览（数字孪生）”一致。
- 技术名词允许保留英文：Kafka、Flink、ClickHouse、OpenSearch、NebulaGraph、MinIO、PCAP、TLS、IP、DNS。
- 数字使用等宽数字或 `font-variant-numeric: tabular-nums;`。

### 3.3 尺寸与圆角

| 项目 | 值 |
|---|---:|
| 应用最小宽度 | `1440px` |
| 高保真输出画布 | `1920x1080 px`，所有页面图、浮层图和规范板一律按此尺寸生成 |
| 参考图用途 | 仅作为业务口径、视觉风格和信息密度基线，不作为输出像素尺寸依据 |
| 左侧单栏宽 | `166px`，以当前 `screen.png` 实测公共 AppShell 为准 |
| 顶部状态栏高 | `80px`，以当前 `screen.png` 实测公共 AppShell 为准 |
| 底部状态栏 | `y=997 / h=83`，以当前 `screen.png` 实测公共 AppShell 为准 |
| 面板间距 | `8px` |
| 面板圆角 | `6px` |
| 按钮圆角 | `4px` |
| 表格行高 | `32px` |
| 小型状态卡高 | `56-72px` |

禁止卡片套卡片。页面区块使用网格和面板分组，只有列表项、弹窗、抽屉可以使用卡片化承载。

## 4. 应用框架

### 4.1 顶部栏

顶部栏从左到右包含：

1. 产品 Logo 和标题。
2. 站点选择：如“主园区”。
3. 时间：如 `2026-06-19 17:30:45`。
4. 风险态势：高风险、分值、状态色。
5. 告警总数、关键告警、采集健康度、数据质量。
6. 快速入口：PCAP 检索、资产检索、规则检索、脚本中心、帮助中心、更多应用。

顶部栏不承载通知铃铛、用户头像、用户菜单、设置或电源动作组；顶部的告警总数和关键告警只作为运行指标，不作为通知中心入口。用户身份、角色、在线状态和用户动作归属左侧底部用户区；通知角标、设置、全局配置和电源归属底部状态栏右侧全局动作区。

顶部 KPI 卡采用紧凑横向信息块，不能做成营销卡片。

### 4.2 左侧导航

左侧采用单栏展开式导航，在同一个侧栏内承载两级菜单层级：

- 第一层：固定业务域图标 + 文案。
- 第二层：当前业务域下的页面菜单。
- 选中态：蓝色竖线、浅蓝图标、深蓝渐变底、角标数量。

全局左侧菜单约束：

- 左侧菜单必须与当前 `screen.png` 的 AppShell 单栏展开式导航保持一致，左侧整体实测约 `166px`，一级业务域和当前业务域二级页面同处一个侧栏。
- 一级菜单固定为：综合态势、采集监测、威胁分析、资产图谱、检测运营、审计配置。
- 二级菜单只显示当前一级业务域下的页面，不显示其他业务域页面。
- 二级菜单项保持紧凑列表形态，项高约 `40px`，圆角 `4px`，细描边或透明底；当前项使用蓝色左侧高亮条和低饱和蓝底。
- 禁止恢复“窄一级栏 + 独立二级栏”的双栏结构。
- 禁止新增第三层菜单，禁止把二级菜单画成厚重卡片、巨大按钮、工具面板或专题目录。
- 禁止把页面内部模块放进左侧菜单，例如告警队列、筛选检索、研判时间线、专题分析 Tabs 等只能放在主内容区。
- 登录/认证页和移动端导航抽屉可以作为明确例外，但视觉 token、菜单命名和层级关系仍需与 foundations 保持一致。

当前 `web/ui/src/components/Layout/MainLayout.tsx` 已具备真实路由分组，应作为前端实现真源。

### 4.3 主工作区

主工作区使用 12 栅格思路：

- 左上：园区拓扑总览。
- 右上：采集与流处理管道健康。
- 左中：告警队列。
- 中中：告警时间线与关联告警簇。
- 下方：流量趋势、流量拓扑、证据列表。
- 右侧固定详情栏：选中告警详情、资产上下文、风险评分、响应处置、反馈与学习、模型状态、验收证据。

业务流向必须清晰体现：

```text
园区拓扑 -> 探针采集 -> Kafka/Flink/存储 -> 告警研判 -> 证据取证 -> 响应处置 -> 反馈学习 -> 验收证据
```

## 5. 组件规范

### 5.1 Panel

Panel 是本系统最小业务容器。

要求：

- 标题左对齐，字号 `15px`，字重 `600`。
- 标题栏高度 `34-38px`。
- 面板边框使用低透明青色，不使用强投影。
- 面板内容必须有固定最小高度或响应式约束，避免图表加载后挤压标题。

建议 CSS：

```css
.taf-panel {
  background: var(--ui-bg-panel);
  border: 1px solid var(--ui-border-subtle);
  border-radius: 6px;
  overflow: hidden;
}

.taf-panel__header {
  height: 36px;
  display: flex;
  align-items: center;
  padding: 0 12px;
  color: var(--ui-text-primary);
  font-size: 15px;
  font-weight: 600;
  line-height: 20px;
}
```

### 5.2 KPI 状态卡

KPI 卡用于顶部和链路健康展示。

要求：

- 标题 `11-12px`，数值 `20px` 或更大。
- 数字必须使用等宽数字。
- 状态色只表达风险等级和健康状态。
- 同一行 KPI 高度一致，不随数值长度跳动。

### 5.3 表格

告警、证据、资产列表采用暗色高密度表格。

要求：

- 表头背景比正文略亮。
- 行高 `32px`，紧凑但不压缩到不可读。
- 状态列使用 Badge/Tag，不只用文字。
- 操作列使用图标按钮，悬停显示 tooltip。

### 5.4 时间线

告警时间线必须展示：

- 时间点。
- 阶段名称。
- 阶段状态。
- 当前选中告警 ID 或追踪信息。

时间线是研判闭环的中心，不应退化为普通日志列表。

### 5.5 图表

ECharts 统一规范：

- 背景透明。
- 轴线与网格线使用 `rgba(120, 190, 220, 0.16)`。
- 文字使用 `--ui-text-secondary`。
- 入站/出站流量使用蓝和青。
- 告警/异常使用红。
- 健康/通过使用绿。
- Tooltip 使用深色背景和青色边框。

### 5.6 右侧详情栏

右侧详情栏是闭环动作入口，不只是信息展示。

固定顺序：

1. 选中告警详情。
2. 资产上下文。
3. 风险评分。
4. 响应处置。
5. 反馈与学习。
6. 模型状态。
7. 验收证据。

响应处置按钮包括隔离主机、阻断 IP、封禁账户、下发脚本。危险动作必须二次确认并写入审计日志。

## 6. 页面落地规则

### 6.0 仪表盘、态势大屏与专题面板的内容分工

仪表盘、态势大屏和专题面板允许共享顶部状态条、全局时间窗、站点选择和底层数据源，但主工作区不能复用同一套组件。视觉差异不能解决内容重叠，必须按业务问题拆分：

| 页面 | 中心对象 | 回答的问题 | 主工作区应使用 | 不应使用 |
|---|---|---|---|---|
| `/dashboard` 仪表盘 | 脱敏待处理事项与运营工作量 | 我今天要优先关注什么？哪些 SLA、证据、反馈、复核和门禁有风险？ | 脱敏运营 KPI、优先级待办队列、采集与数据健康门禁、告警处置阶段工作篮、证据/反馈质量摘要、验收缺口看板 | 责任人、负责人、头像、班组、值班表、个人账号、联系方式、待指派、未认领责任、交接名单、个人任务归属、大尺寸园区拓扑、横向全链路管道、攻击热力地图、流向动画 |
| `/screen` 态势大屏 | 系统闭环叙事 | 系统整体如何从全流量采集闭环到研判、取证、响应、反馈和验收？ | 园区拓扑、采集流处理管道、威胁态势、证据完整度、响应反馈闭环、运行底座 | 高密度任务表、批量操作、人员矩阵、复杂筛选、工单逾期清单 |
| `/topics` 专题面板 | 固化的攻击场景与证据范围 | 加密隧道、数据外传或 APT 战役现在怎么样？证据够不够？能否形成报告？ | 页内专题切换、场景化 KPI、范围标签、保存视图、局部影响面、专题分析、关联证据、报告导出 | 三个左侧专题菜单、全局值班待办、交接班、全局大屏拓扑、全链路管道、系统配置 CRUD、通用专题目录 |

落地规则：

1. 三个页面可以共用指标口径，但不能共用同一张完整表格、同一组完整图表或同一套右侧详情栏。
2. 仪表盘展示“异常原因 + 风险优先级 + 下一步动作类型”，每个模块都应能下钻到告警、数据质量、取证、审计或合规页面。
3. 态势大屏展示“整体状态 + 闭环路径”，适合只读 token、脱敏演示和自动轮播。
4. 专题面板展示“范围 + 场景 + 证据 + 报告”，当前以单个 `/topics` 页面承接，加密隧道、数据外传和 APT 战役只能作为页内切换；旧 `/topics/tunnel`、`/topics/exfil`、`/topics/apt` 仅作为兼容深链，不进入左侧菜单。
5. 任何筛选、批量操作、编辑、确认、下载、审批类内容优先放仪表盘、专题面板或业务页，不放态势大屏。
6. 任何远距离理解、汇报展示、链路叙事和演示脱敏内容优先放态势大屏，不放仪表盘主区。
7. 专题面板可以有报告导出和订阅/静默，但不承担全局工单调度和交接班职责。

### 6.1 `/screen`

定位：态势感知与指挥研判大屏。

必须优先对齐视觉参考图，形成一屏闭环。适用于展示、值班墙、售前汇报和验收演示。主画面应以园区拓扑、采集流处理管道、威胁态势、证据完整度、响应反馈和运行底座为核心；待办、SLA 和复核只保留汇总状态，不展开成工作台，也不出现人员归属信息。

### 6.2 `/dashboard`

定位：日常运营工作台。

继承视觉参考图的顶部 KPI、采集健康和深色工作台风格，但主内容必须从“态势展示”转为“脱敏运营工作台”。应围绕超时 SLA、临近超时数、高危未处理、待取证、待反馈、待复核、队列积压、健康门禁、证据/反馈质量和验收缺口组织，不放大尺寸园区拓扑、横向全链路管道或攻击热力地图。

敏感信息约束：仪表盘主区不得展示责任人、负责人、头像、班组、值班表、个人账号、联系方式、待指派、未认领责任、交接名单或个人任务归属。确需查看人员或组织归属时，应跳转到具备权限控制和审计记录的业务详情页。

### 6.3 `/topics`

定位：场景化专题视角、保存视图和报告交付入口。

专题面板应把“从告警、资产、业务系统、园区、检测场景或验收指标筛出来的一组对象”固化为可复用视图。当前现役前端只保留一个左侧菜单入口：`/topics` 专题面板；加密隧道、数据外传、APT 战役是专题面板内的切换状态，并分别读取 `/v1/topics/tunnel`、`/v1/topics/exfil`、`/v1/topics/apt`。旧 `/topics/tunnel`、`/topics/exfil`、`/topics/apt` 只保留兼容深链，进入后映射到 `/topics?topic=...`。

专题面板内的每个专题状态必须包含场景化范围、专题专属 KPI、局部影响面、专题分析、关联事件与证据、跨域下钻、报告预览和导出入口。不得做成小号仪表盘，也不得复用态势大屏的全局拓扑和全链路管道。

交互规则：

1. 专题级动作使用浮层：保存视图、编辑专题范围、报告导出、证据包导出、试点周报导出、订阅、静默、分享和收藏。
2. 业务级详情必须跳转：告警、战役、资产、PCAP、Session、图谱、加密流量、合规证据和审计记录不在专题页内复制详情页。
3. 从专题跳转业务页时必须携带 `topicId`、时间窗、资产组、检测场景、攻击阶段和筛选条件；业务页应保留“返回专题”入口。
4. 专题浮层只编辑专题元数据、范围和报告参数，不执行隔离、阻断、封禁、模型发布、规则发布等危险业务动作。

Tab 拆分设计合并规则：`/topics` 已有加密隧道、数据外传、APT 战役三张拆分设计输入，前端实现必须合并为同一个 `/topics` 页面内的 Tab/Segmented 状态，不新增左侧菜单或独立业务路由。后续其他页面如果 UI 设计图按 Tab 拆成多张，也按同样规则合并到单一路由页面内；只有产品文档明确要求新增导航入口时，才允许拆出新的菜单或路由。

### 6.4 `/alerts` 和 `/alerts/:alertId`

定位：威胁研判主链路。

必须包含告警列表、时间线、证据、资产上下文、响应动作、反馈学习。详情页可以采用三栏布局，但视觉 token 必须与参考图一致。

### 6.5 `/forensics`、`/graph`、`/assets`

定位：证据与关系追踪。

必须支持从告警上下文进入，保持告警 ID、资产、IP、时间窗和证据链不丢失。

### 6.6 `/data-quality`、`/compliance`、`/audit-log`

定位：工程验收和审计交付。

采用同一深色规范，但减少装饰性流光，突出证据完整度、P0/P1 门禁、审计链路、通过/失败原因。

## 7. 前端实现建议

### 7.1 Token 落地

建议在 `web/ui/src/styles/global.css` 增加 `--ui-*` token，并逐步替换现有 `--bg-*`、`--accent-*` 变量。

### 7.2 组件抽象

建议新增或统一以下组件：

| 组件 | 用途 |
|---|---|
| `AppShell` | 顶部栏、左侧单栏展开式导航、底部状态栏 |
| `TafPanel` | 标准面板容器 |
| `KpiTile` | 顶部和链路 KPI |
| `SeverityBadge` | 高危/中危/低危 |
| `HealthBadge` | 正常/异常/离线/通过 |
| `EvidenceTable` | PCAP、会话、日志证据表 |
| `AlertTimeline` | 告警研判阶段时间线 |
| `ResponseActionBar` | 响应处置动作组 |
| `FeedbackBlock` | 反馈标注和模型学习入口 |

### 7.3 Ant Design 约束

- 继续使用 Ant Design，但必须通过全局 token 覆盖暗色主题。
- 表格、抽屉、弹窗、下拉、选择器和 Tabs 必须统一深色样式。
- 按钮不使用大圆角；默认圆角 `4px`，面板圆角 `6px`。
- 表单项标签和表格表头不能使用默认浅色背景。

### 7.4 ECharts 约束

- 每张图必须有 loading、empty、error 状态。
- 图例不能与标题重叠。
- 图表容器必须有稳定高度，避免加载后布局跳动。
- 2K/4K 大屏下曲线宽度和字体需要按容器缩放，不按 viewport 盲目放大。

## 8. 验收清单

前端实现每个页面至少满足：

- 面板标题全部中文-only，字号一致。
- 一级菜单和二级菜单与 `MainLayout.tsx` 路由分组一致。
- 左侧菜单与当前 `screen.png` 的单栏展开式 AppShell 一致，不恢复独立二级栏，不出现第三层菜单、卡片式二级菜单或页面内部模块菜单化。
- 高危/中危/低危/健康/通过颜色一致。
- 表格、图表、右侧详情栏在 1440x900、1920x1080、2560x1440 下不遮挡。
- 数据加载中、空数据、错误、无权限状态均有设计。
- 告警到证据、资产、响应、反馈、审计能闭环跳转。
- Playwright 截图巡检无文字溢出、无图表遮挡、无 console error。
