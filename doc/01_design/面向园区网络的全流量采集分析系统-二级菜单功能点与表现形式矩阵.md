# 面向园区网络的全流量采集分析系统二级菜单功能点与表现形式矩阵

更新时间：2026-06-26

## 1. 编制目标

本文档承接《课题一产品与技术总体设计》《左侧菜单信息架构》《UI 前端规范》和《UI 设计套装》，把左侧 6 个一级菜单下 24 个二级菜单进一步拆成可设计、可开发、可验收的页面功能点。

本矩阵解决三个问题：

1. 每个二级菜单页面应该放什么业务内容。
2. 每个页面应具备哪些功能点和闭环动作。
3. 页面内容应采用什么表现形式，便于后续 GPT 生图、Figma 设计和前端实现保持一致。

## 2. 通用页面规则

所有二级菜单页面都必须遵守以下规则：

| 规则 | 要求 |
|---|---|
| 业务闭环 | 页面不能只展示数据，必须有下一步动作、下钻入口或审计留痕 |
| 上下文传递 | 从告警、资产、PCAP、图谱、模型、合规进入其他页面时必须保留时间窗、租户、资产、告警 ID、证据 ID 等上下文 |
| 真实链路 | UI 文案和指标应对应 Probe、Kafka、Flink、ClickHouse、OpenSearch、NebulaGraph、MinIO、PostgreSQL、MLOps 等真实链路 |
| 表现统一 | 深色高密度安全运营台视觉，标题、表格、图表、标签、状态色遵循 UI 前端规范 |
| 左侧菜单一致 | 左侧菜单必须与当前 `screen.png` 的 AppShell 单栏展开式导航一致，一级固定六个业务域，二级在同一侧栏内只显示当前业务域页面；禁止恢复独立二级栏、第三层菜单、卡片式二级菜单和把页面内部模块菜单化 |
| 状态可解释 | 任何红色、黄色、高危、异常、失败状态都必须能说明原因，并能跳转到证据或处理入口 |
| 审计可追溯 | 高风险动作、配置变更、导出下载、模型发布、规则发布、白名单生效都必须进入审计日志 |
| 页面职责去重 | 相邻页面可以共享数据源和指标口径，但不能复用同一套主工作区组件；仪表盘回答“我今天要处理什么”，态势大屏回答“系统整体闭环如何运转”，专题面板回答“某个攻击场景和证据范围是否足以研判和报告” |

常用表现形式：

| 表现形式 | 适用场景 |
|---|---|
| 顶部 KPI 条 | 风险态势、告警总量、采集健康、数据质量、SLA、存储容量 |
| 拓扑图 / 数字孪生 | 园区、链路、探针、资产、实体关系、攻击路径 |
| 时间线 | 告警研判、攻击阶段、任务执行、证据生成、审计轨迹 |
| 高密度表格 | 告警、资产、探针、证据、规则、模型、任务、审计日志 |
| 图表 | 折线、堆叠面积、环图、桑基图、雷达图、热力图、箱线图、分布图 |
| 详情抽屉 | 告警、资产、实体、规则、模型、证据等对象详情 |
| 弹窗 | 创建、编辑、确认、批量操作、危险动作二次确认 |
| 右侧闭环栏 | 响应处置、反馈学习、证据状态、模型状态、验收状态 |
| 状态机 | 告警、战役、部署、训练、剧本、白名单审批、验收门禁 |

## 3. 综合态势

### 3.1 仪表盘 `/dashboard`

页面定位：安全值班和日常运营首页，用于回答“今天系统是否健康、风险在哪里、我下一步该优先处理什么”。仪表盘的中心对象是“脱敏的待处理事项和运营工作量”，不是园区拓扑、全链路展示墙或人员责任看板。

内容边界：

- 保留：脱敏运营 KPI、优先级待办队列、SLA、复核、取证任务、反馈任务、健康门禁、证据/反馈质量、验收缺口和审计留痕缺口。
- 降级：总体风险和采集健康只保留为可点击摘要，用于引导下钻。
- 禁止：责任人、负责人、头像、班组、值班表、个人账号、联系方式、待指派、未认领责任、交接名单、个人任务归属、大尺寸园区拓扑、横向全链路管道、攻击热力地图、流向动画、领导汇报式闭环大图。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 脱敏运营 KPI | 超时 SLA、临近超时数、高危未处理、待取证、待反馈、待复核、队列积压量、今日闭环进度 | 紧凑 KPI 条、倒计时、趋势箭头 | 跳转告警中心、取证分析、MLOps 编排、审计日志 |
| 优先级待办队列 | 高危未处理、临近超时、待取证、待反馈、待复核、需审计留痕 | 分组队列表格、优先级标签、SLA 倒计时；列只展示事件 ID、风险级别、资产组、业务系统、处置阶段、剩余时间和证据状态 | 进入告警详情、取证任务或反馈表单 |
| 采集与数据健康门禁 | Probe、Kafka、Flink、ClickHouse、OpenSearch、NebulaGraph、MinIO、PostgreSQL 的通过/警告/失败状态、失败原因、影响范围和最近更新时间 | 门禁矩阵、失败原因 tooltip、影响范围标签 | 下钻数据质量、探针管理、部署管理 |
| 告警处置阶段工作篮 | 今日必处理、处理中、待反馈、待取证、待复核、需审计留痕的告警工作项 | 工作篮列表、分段状态、批量入口 | 更新状态、创建取证任务、补审计 |
| 证据与反馈质量摘要 | PCAP/Session/日志证据完整度、反馈覆盖率、误报回流量、审计留痕缺口、样本回流缺口 | 质量条、缺口清单、趋势小图 | 进入取证分析、反馈学习、审计日志 |
| 验收缺口看板 | 待补证据数、待回流样本数、审计留痕缺口、工单逾期数、合规门禁缺口及建议下一步动作类型 | 缺口看板、红黄绿状态、动作类型列表 | 跳转合规审计、数据质量、部署管理 |

### 3.2 态势大屏 `/screen`

页面定位：领导汇报、售前演示、值班大屏和验收展示，强调“一屏讲清全流量采集分析闭环”。态势大屏的中心对象是“系统闭环叙事”，不是值班待办队列。

内容边界：

- 保留：园区拓扑、探针覆盖、采集流处理管道、攻击阶段、战役簇、证据完整度、响应反馈和运行底座。
- 降级：待办、SLA、复核和验收缺口只显示为总体状态，不展开队列。
- 禁止：高密度任务表、批量操作、责任人、负责人、头像、班组、值班表、交接名单、工单逾期清单、复杂筛选表单。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 园区拓扑总览 | 楼宇覆盖率、园区在线覆盖、核心链路状态、汇聚链路状态、异常链路位置、探针覆盖地图 | 大尺寸数字孪生拓扑、链路动画、状态图例 | 跳转仪表盘或探针管理 |
| 采集流处理管道 | 探针采集、协议解析、归一化、Kafka、Flink、ClickHouse、OpenSearch、NebulaGraph、MinIO | 横向管道、节点健康灯、吞吐水位、流向动画 | 下钻数据质量、部署管理 |
| 威胁态势 | 攻击阶段热度、战役簇密度、风险区域密度、异常链路影响面、外联流向强度 | 攻击阶段条、告警簇、热力地图、桑基流向 | 跳转告警中心、战役列表、攻击链分析 |
| 证据取证 | PCAP 覆盖率、Session 还原率、日志关联率、对象存储归档率、hash 校验通过率、签名 URL 可用率 | 证据完整度条、证据文件卡、取证状态带 | 跳转取证分析 |
| 响应与反馈 | 隔离动作数、阻断动作数、封禁动作数、下发脚本数、反馈标注数、模型学习批次数 | 右侧闭环栏、动作状态、反馈学习环 | 跳转 SOAR 剧本、MLOps 编排 |
| 运行底座 | 大屏刷新间隔、拓扑渲染延迟、链路带宽水位、流向动画帧率、展示脱敏状态 | 底部状态栏、运行状态芯片 | 跳转系统设置、合规审计 |

### 3.3 专题面板 `/topics`

页面定位：围绕攻击场景和证据范围组织的场景化专题视角。当前现役专题统一收敛到单个二级菜单“专题面板”：`/topics`。加密隧道专题、数据外传专题、APT 战役专题是页面内的三种业务切换状态，不作为左侧菜单项。专题面板的中心对象不是“全局态势”或“今日待办”，而是一个被固化的分析范围：一组筛选条件、一类检测场景、一批资产/部门/业务系统，或一个可持续跟踪并可导出材料的交付主题。旧 `/topics/tunnel`、`/topics/exfil`、`/topics/apt` 只保留兼容深链，进入后映射到 `/topics?topic=...`。

内容边界：

- 保留：专题范围、保存视图、专题专属 KPI、局部影响面、专题分析、关联告警/战役/证据、跨域下钻、报告预览、订阅和导出。
- 降级：总体风险、采集健康和待办事项只作为当前专题内的摘要，不展开为全局仪表盘。
- 禁止：全局值班待办队列、交接班、负责人矩阵、大尺寸园区总拓扑、横向全链路管道、领导汇报式闭环大图、系统配置 CRUD。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 专题入口分工 | 左侧二级菜单只有“专题面板”一个入口；加密隧道专题、数据外传专题、APT 战役专题在 `/topics` 页内切换 | 单个左侧菜单、页内专题切换控件、面包屑、跨域下钻入口 | 在三个专题间切换，保留时间窗和场景上下文 |
| 专题范围定义 | 站点、部门、资产组、业务系统、IP 段、账号、协议、规则、模型、攻击阶段、时间窗、来源页面 | 范围标签、条件构造器、保存视图条、来源上下文条 | 从告警筛选固化为专题；切换专题范围并刷新数据 |
| 专题专属 KPI | 专题资产数、活跃探针数、专题流量、关联告警数、高危资产数、关联战役数、证据完整度、报告就绪度、未闭环风险数 | 紧凑 KPI、同比环比、状态色、可点击数字 | 跳转专题内告警、资产、取证、合规和报告 |
| 局部影响面 | 专题范围内资产、探针、链路、外联目的地、账号、服务、告警和战役关系 | 局部拓扑、关系图、流向图、影响矩阵 | 下钻实体图谱、资产台账、攻击链分析 |
| 专题分析 | 隧道协议和高频源、数据外传源和路径、APT 战役阶段、园区/部门/业务系统风险、验收门禁差距 | 分页 Tabs、趋势图、排行表、攻击阶段时间线、风险类型标签 | 进入加密流量、战役列表、告警中心、合规审计 |
| 关联事件与证据 | 专题关联告警、战役、PCAP、Session、日志、图谱路径、审计记录 | 高密度关联表、证据文件卡、完整度条、hash 标签 | 进入告警详情、取证分析、审计日志，补齐证据 |
| 专题报告与交付 | 风险摘要、趋势、处置进度、证据清单、模型/规则版本、试点周报、验收材料 | 报告预览、导出弹窗、报告就绪度、订阅/静默入口 | 导出 PDF/Word/证据包；生成合规/试点/售前材料 |

交互边界：

| 类型 | 使用方式 | 适用内容 |
|---|---|---|
| 专题专属浮层 | 使用 Modal、Drawer、Dropdown 或 Popconfirm，底层仍保持专题页可见 | 保存专题视图、编辑专题范围/名称/刷新周期、专题报告导出、证据包导出、试点周报导出、订阅、静默、分享、收藏 |
| 业务详情跳转 | 跳转到对应业务页面，并携带 `topicId`、时间窗、资产组、攻击阶段和筛选条件 | 告警详情、战役详情、PCAP/Session/证据、资产详情、实体图谱、加密流量、合规审计、审计日志 |

专题面板不得复制告警详情、取证详情、资产详情、图谱详情和合规证据详情。它只负责摘要、编排、报告和上下文传递。

## 4. 采集监测

### 4.1 探针管理 `/probes`

页面定位：管理园区全流量采集入口，证明系统“看得见、采得全、采得稳”。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 探针总览 | 探针数量、在线率、版本、采集模式、CPU/内存、接口状态 | KPI 卡、探针状态矩阵 | 跳转探针详情 |
| 部署拓扑 | 探针与楼宇、交换机、链路、镜像口、采集网卡关系 | 拓扑图、地图点位、链路高亮 | 下钻园区拓扑和资产台账 |
| 吞吐与丢包 | PPS、Gbps、丢包率、解析率、背压、批量发送状态 | 实时折线、阈值线、异常标记 | 跳转数据质量 |
| 探针配置 | 采集接口、过滤策略、PCAP 归档、mTLS、CPU 亲和、缓冲区 | 配置表单、版本对比、变更抽屉 | 下发配置并写审计 |
| 心跳与日志 | 最近心跳、异常日志、重启记录、证书状态 | 时间线、日志列表、健康标签 | 触发重启、升级、证书轮换 |
| 批量运维 | 批量升级、批量启停、策略下发、连通性测试 | 批量操作栏、确认弹窗 | 进入部署管理和审计日志 |

### 4.2 数据质量 `/data-quality`

页面定位：证明数据可信，定位采集、解析、传输、处理、入库中的质量问题。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 质量总分 | 完整性、及时性、准确性、重复率、字段缺失、DLQ 数量 | 质量评分、雷达图、红黄绿门禁 | 跳转合规审计 |
| Topic 健康 | Kafka offset、积压、消费延迟、分区倾斜、消息大小 | Topic 表格、延迟折线、分区热力 | 定位 Flink 作业或探针 |
| Flink 处理质量 | checkpoint、watermark、backpressure、异常、迟到数据 | 作业健康卡、状态时间线 | 跳转部署管理 |
| 字段质量 | 五元组、community_id、tenant、asset_id、protocol、timestamp 缺失和异常 | 字段矩阵、样例抽屉 | 生成修复任务或重放任务 |
| 存储质量 | ClickHouse 写入、OpenSearch 索引、NebulaGraph 边写入、MinIO 对象归档 | 存储健康卡、写入速率、容量趋势 | 进入系统设置或部署管理 |
| 重放与对账 | DLQ 重放、时间窗对账、幂等检查、重复检测 | 重放任务表、对账报告 | 形成验收证据 |

## 5. 威胁分析

### 5.1 告警中心 `/alerts`

页面定位：安全研判主工作台，围绕告警完成筛选、研判、取证、处置、反馈。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 告警队列 | 高危、中危、低危、未处理、处理中、已确认、已忽略 | 高密度表格、严重级别标签、批量选择 | 批量指派、批量状态变更 |
| 筛选检索 | 时间窗、资产、IP、规则、模型、攻击阶段、状态、置信度 | 筛选条、搜索框、保存视图 | 固化为对应专题页 |
| 选中告警摘要 | 告警 ID、名称、风险评分、资产、源/目的、规则/模型 | 右侧详情栏、风险评分环 | 进入告警详情 |
| 研判时间线 | 首次发生、异常行为、横向移动、证据生成、处置动作 | 时间线、阶段节点、证据锚点 | 跳转取证分析 |
| 关联告警簇 | 同源 IP、同资产、同攻击链、同规则、同战役 | 关联图、聚类卡片 | 合并战役或进入战役详情 |
| 处置与反馈 | 隔离、阻断、封禁、下发脚本、TP/FP、误报原因、白名单草案 | 操作按钮、反馈表单、确认弹窗 | 写入 SOAR、MLOps、审计日志 |

### 5.2 战役列表 `/campaigns`

页面定位：把多条告警聚合成事件/战役，支持跨时间、跨资产、跨阶段的调查。

| 功能点 | 内容 | 表现形式 | 闭环动作 |
|---|---|---|---|
| 战役总览 | 战役数量、活跃战役、影响资产、最高风险、持续时间 | KPI 卡、趋势图 | 跳转战役详情 |

## 10. 页面设计落地检查清单

后续生成 GPT UI 图、Figma 原型或前端页面时，每个二级菜单都需要检查：

1. 页面是否有明确的业务定位，而不是泛用 CRUD。
2. 页面是否至少包含一个可下钻对象：告警、资产、探针、证据、规则、模型、任务或审计记录。
3. 页面是否展示关键状态的原因，而不是只用颜色标红。
4. 页面是否有明确的下一步动作：处理、取证、发布、回滚、反馈、导出或审计。
5. 页面是否保留跨页上下文：时间窗、租户、资产、告警 ID、证据 ID、规则/模型版本。
6. 页面是否遵循 UI 前端规范的字体、标题、状态色、卡片密度和右侧闭环栏规则。
7. 危险动作是否有二次确认、权限校验、影响范围说明和审计日志。
8. 页面是否能映射到课题一验收指标或工程运行证据。
