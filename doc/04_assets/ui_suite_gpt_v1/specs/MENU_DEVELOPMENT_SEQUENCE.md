# 菜单与 Tab 同步开发顺序

更新时间：2026-07-17

## 固定开发顺序

1. 加密流量 `/encrypted-traffic`
2. 取证分析 `/forensics`
3. 探针管理 `/probes`
4. 数据质量 `/data-quality`
5. 规则管理 `/rules`
6. 部署管理 `/deployments`
7. 模型管理 `/models`
8. MLOps 编排 `/mlops`
9. 其他菜单

不得为了先关闭容易页面而改变上述顺序。前一菜单只有在页面级主状态、全部 Tab 状态、后端/API、持久化数据、业务交互、真实浏览器截图和审查回修均形成证据后，才能进入下一菜单。

## Tab 同步开发规则

- 菜单页与其所有 Tab 属于同一个开发工作包，不允许只完成默认 Tab。
- 所有 Tab 必须保留在所属菜单的原 Tab 容器和原菜单路由内切换；禁止把 Tab 改造成独立页面、独立路由或新增 `view` 查询参数。
- 每个 Tab 必须拥有独立内容契约、loading/error/empty 状态和同名验收截图，但验收路由必须始终是所属菜单路由。
- 切换 Tab 保留公共 AppShell、菜单标题、Tab 栏和公共筛选条件；业务模块不能仅更换标题后复用同一内容，Tab 栏不得滚动或被业务内容挤出可视区域。
- Tab 内存在子 Tab、Drawer、Modal 或详情状态时，必须随所属 Tab 一并实现和验收。
- UI、API、数据库/ClickHouse/Kafka 数据口径必须一致；API 空数据不得由生产页面静默替换成演示数据。

## PCAP 回放数据策略

数据根目录：`/home/wangwt/task/datasets`

- 先使用小体积、标签明确的 PCAP 做解析和回放冒烟，再使用代表性加密、隧道、外联和攻击样本做业务回归。
- 数据集只作为只读输入，不移动、不改名、不覆盖原文件。
- 大于 1 GiB 的样本必须经过显式资源计划后再回放，禁止直接全量灌入共享环境。
- 每次回放必须记录文件路径、大小、SHA-256、标签、回放速度、读取包数、解析包数、运行 ID 和下游证据。
- 页面验收只接受 PCAP 回放后经 Probe/Kafka/Flink/ClickHouse/API 到达 UI 的数据，或明确标记为 empty/partial；不接受无标识的前端固定数组。

## 单菜单强化闭环

`UI 拆解 -> Tab 契约 -> 前端实现 -> 后端实现 -> 数据库/流数据 -> PCAP 回放 -> Windows Chrome -> diff/metrics -> 逻辑审查 -> 布局审查 -> 主线程裁决 -> 回修`

每轮失败 finding 进入下一轮修复输入；不得通过降低无业务依据的阈值获得假绿。
