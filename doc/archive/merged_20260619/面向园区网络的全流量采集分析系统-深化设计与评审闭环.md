## 1. 结论先行

本系统应被定义为“面向园区的全流量安全运营底座”，不是单一抓包工具、单点 IDS、传统 SIEM 或仅用于展示的大屏。它的价值链是：

> 看得见、判得准、查得到、改得动、验得过。

“看得见”来自 Rust Probe、Ingest Gateway、Kafka、Flink 和多源数据融合；“判得准”来自规则、行为模型、CEP、攻击链、资产和用户上下文；“查得到”来自 Evidence、PCAP 索引/裁剪、MinIO 对象、实体图谱和审计；“改得动”来自白名单、规则灰度、模型注册/激活和 MLOps 反馈；“验得过”来自可复核的任务书指标矩阵、真实链路冒烟、专项压测、第三方测试和试点材料。

当前工程状态适合进入“设计完善 + 验收证据补齐”阶段。根据已合并到 `doc/05_status/代码实证状态核对-2026-06-19.md` 和 `agent.md` 的真实链路证据，页面/API/多条 Kafka-Flink-DB-API-UI 链路已经具备真实联调证据；剩余关键缺口是 10 x 100Gbps/512Mpps 专项压测、端到端 P95 <= 60s 分布报告、95% 准确率与误报率低于 5% 的大样本/第三方评测、模型泛化质量、生产级 Kafka TLS/SASL 与密钥治理。

## 4. 产品设计完善

### 4.1 产品定位

系统面向高教园区、科研园区、产业园区和政企园区，解决“高带宽下看不清、告警多而不准、证据难回放、规则模型难持续优化、验收材料难闭环”的问题。

边界要写清楚：

- In scope：全流量采集、会话化、特征工程、检测告警、攻击链、加密流量、数据融合、行为基准、PCAP 取证、实体图谱、资产、日志、用户行为、规则/模型/白名单/部署治理、审计与合规报告。
- Out of scope：完整 EDR、工单系统替代、跨域 SOC 全量运营、未经审批的自动阻断。
- 与其他课题协同：课题一输出风险画像、证据链、检测结论和策略建议；课题二/四/五承接防护执行、VPN 专项识别、关键设备或网关联动，再把处置结果和误报反馈回流课题一。

### 4.2 目标用户与 Jobs To Be Done

| 角色 | 要完成的事 | 系统必须给出的闭环 |
|---|---|---|
| 安全值班员 | 在高告警量中快速发现真实高危事件 | 态势总览、筛选、严重级别、采集健康、Top 资产 |
| 安全研判员 | 把告警判成真实威胁、误报或需持续观察 | 告警详情、证据、PCAP、图谱、反馈、状态机 |
| 平台管理员 | 稳定管理探针、规则、模型、白名单和部署 | 探针心跳、规则版本、灰度、回滚、审计 |
| 项目验收人员 | 判断课题一指标是否可考核 | 指标矩阵、测试报告、样本集、演示脚本 |
| 园区负责人 | 向业务方解释风险、处置和投入价值 | 合规报表、趋势、MTTR、误报率、试点效果 |

### 4.3 竞争力设计

| 对比对象 | 传统弱点 | 本系统竞争点 | 证明材料 |
|---|---|---|---|
| SIEM | 偏日志汇聚，缺原始网络证据 | 全流量遥测 + PCAP 证据 + 图谱追踪 | Flow/Session/PCAP/Graph 链路 |
| IDS | 单点规则命中，误报沉淀弱 | 规则 + CEP + 行为模型 + 反馈学习 | Rule/Behavior/CEP/MLOps |
| NDR | 泛化场景较宽，园区交付材料弱 | 园区多源融合、试点验收和课题指标绑定 | 任务书矩阵、试点清单 |
| 抓包平台 | 查包依赖专家，闭环弱 | 从包到结论，从结论到规则/模型优化 | Evidence、Forensics、Feedback |

业务校准参考了 NIST CSF、CISA Zero Trust Maturity Model、MITRE ATT&CK 数据源和 Arkime 全包取证方向。共同趋势是：高质量遥测、持续监测、证据可回放、响应闭环和可复核改进。

## 6. 技术设计深化

### 6.1 主链路

```text
Probe Agent
  -> Ingest Gateway
  -> Kafka
  -> Flink Session/Feature/Rule/Behavior/CEP/Alert/Log/User Jobs
  -> ClickHouse/PostgreSQL/OpenSearch/NebulaGraph/Redis/MinIO
  -> Go control-plane APIs
  -> Web UI
  -> Feedback/Whitelist/Rule Review/MLOps
  -> Kafka rule.updates/model-updates
  -> Flink 热更新
```

这条链路的工程设计原则是契约优先、流批结合、安全内建、证据可追踪。`proto/traffic/v1` 是跨语言真源；Kafka Topic、DDL、REST API 和 TypeScript 类型必须同步维护。

### 6.3 安全生产化

### 6.4 性能压测闭环

| 指标 | 口径 | 采样点 | 验收材料 |
|---|---|---|---|
| 10 x 100Gbps / 512Mpps | 多口并发、持续压测、丢包率、CPU/NUMA/内存水位 | NIC、Probe、Ingest、Kafka、Flink | 压测报告、监控截图、流量生成配置 |
| P95 <= 60s | `ui_seen_ts - event_ts`，分位数而非均值 | `event_ts`、`ingest_ts`、`kafka_ts`、`flink_out_ts`、`alert_ts`、`api_ts`、`ui_seen_ts` | SQL/PromQL、端到端延迟分布 |

### 6.5 检测质量闭环

第三方测试前必须准备：

1. 样本集：已知攻击、未知攻击、正常业务、加密/隧道、横向移动、数据外传、BOT、扫描爆破。
2. 标签：样本 ID、攻击类型、ATT&CK 阶段、源/目的资产、时间窗、人工复核人。
3. 公式：Accuracy、Precision、Recall、F1、FPR、FNR、误报率、漏报率。
4. 报告：混淆矩阵、按攻击类型分层、按数据源缺失分层、版本号、规则/模型版本。
5. 回归：每次规则/模型发布必须跑固定回归集，生成差异报告。

## 7. 告警到学习的闭环设计

告警运营不是“列表里点关闭”，而是一个可审计的学习环：

1. Alert Generator 生成告警，Alert Service 去重、落库、索引证据。
2. 研判员在详情页查看 Evidence、PCAP、Graph、Campaign、资产上下文。
3. 判定 TP/FP/需观察，写入反馈和原因码。
4. FP 进入白名单草稿或规则复审；TP 进入关闭归档、报告和样本库。
5. 反馈数据进入 `alert.feedback.v1` 和 ClickHouse。
6. MLOps preflight 判断反馈量、漂移、样本质量和训练条件。
7. 训练、评估、注册、激活后发布 `model-updates`。
8. Behavior Job 广播新模型，灰度或回滚后刷新指标。

核心门禁：没有样本质量、回归结果和回滚策略的模型不得自动全量激活。

## 9. 多轮评审结果

本轮按用户要求组织角色化评审。需要明确：`multi_role_review_ledger.csv` 只证明评审规模和角色覆盖，不单独代表专家深度；深评结论以 `doc/review/专家深评整改清单.md` 为准，并已把子代理返回的可执行问题转成整改项。

| 阶段 | 角色 | 次数 | 产出 |
|---|---|---:|---|
| 初评 | 产品经理、技术经理、销售经理 | 1000 | 产品/工程/售前初评结论 |
| 中期评审 | 产品经理、技术经理、销售经理、项目经理、全栈工程师、算法工程师、实施工程师、测试工程师、UI工程师 | 10000 | 九角色批量评审台账 |
| 技术总监复审 | 技术总监 | 100 | 架构、生产化、验收门禁复审 |
| 合计 | 10 个角色 | 11100 | `doc/review/multi_role_review_ledger.csv` |

高频结论：

- 产品经理：功能闭环强，考核闭环要补样本、标签、公式、第三方测试流程。
- 技术经理：架构链路完整，性能、P95、准确率/误报率、DLQ 重放、HA 演练和生产安全是核心缺口。
- 销售经理：卖点应从技术栈转为客户价值，“从包到结论，再到优化”是对 SIEM/IDS/NDR/抓包平台的差异化。
- 中期九角色：要把 PRD、SDD、代码、部署、测试、试点材料和演示脚本绑定成同一套验收口径。
- 技术总监：验收前必须声明基线版本，任何性能和模型指标都要以报告为准，不能以“架构具备”替代“实测达成”。

## 10. 验收门禁

| 门禁 | 当前判断 | 必须材料 |
|---|---|---|
| 功能主链路 | 已具备强证据 | live smoke、真实 API、页面巡检、E2E 报告 |
| 性能吞吐 | 待闭环 | 10 x 100Gbps/512Mpps 压测 |
| 端到端延迟 | 待闭环 | P95 <= 60s 分布报告 |
| 检测质量 | 待闭环 | 第三方测试、样本集、标签、混淆矩阵 |
| 模型泛化 | 待闭环 | 大样本评估、漂移、回滚、灰度 |
| 生产安全 | 待闭环 | Kafka TLS/SASL、Secret/ExternalSecret、证书轮换 |
| HA/故障恢复 | 待闭环 | Kafka/CH/Flink/PG/MinIO 故障演练 |
| 试点应用 | 待闭环 | 部署证明、用户盖章、经济效益材料 |

门禁口径不得混用：100 轮 live smoke 可以写成“功能回归强证据”，不能写成“性能、检测质量、第三方验收已通过”。10 x 100Gbps、512Mpps、P95 <= 60s、95%/5% 和生产安全必须分别出专项报告。

检测质量门禁必须从“单个 F1 值”升级为任务书口径：accuracy >= 95%、FPR/误报率 < 5%、unknown recall、最小样本量、95% 置信区间、阈值约束、离在线一致性、特征覆盖率和回放可复现。小样本 MLOps 成绩只能写成工程闭环证明，不能写成泛化质量证明。

## 11. 外部业务闭环参考

- [NIST Cybersecurity Framework 2.0](https://www.nist.gov/cyberframework)：强调 Govern、Identify、Protect、Detect、Respond、Recover 的连续治理闭环。
- [CISA Zero Trust Maturity Model](https://www.cisa.gov/zero-trust-maturity-model)：强调 identity、devices、networks、applications、data 的可见性、分析和自动化。
- [MITRE ATT&CK Data Sources](https://attack.mitre.org/datasources/)：网络流量、认证日志、进程/资产/用户上下文是攻击行为检测的重要数据源。
- [Arkime](https://arkime.com/)：全包捕获、索引和检索是网络取证闭环的重要能力，但需要和告警、资产、图谱、反馈联动才能形成产品竞争力。
