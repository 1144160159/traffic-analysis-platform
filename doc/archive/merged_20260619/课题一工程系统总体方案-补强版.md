# 课题一工程系统总体方案（补强版）

更新时间：2026-06-18  
对象：园区网络流量智能检测与分析  
定位：从任务书指标到工程系统、产品体验、验收证据的一体化方案。

## 1. 建设目标

课题一的工程系统不应只解释“用了哪些组件”，而应证明一件事：园区网络中的流量、资产、设备日志、用户行为能够被稳定采集、融合、分析、研判、取证和反馈优化，并最终支撑“检测到已知和未知网络攻击类型，预警准确率达到 95% 以上，误报率低于 5%”的第三方可测目标。

本系统的工程目标分为四层：

| 层次 | 目标 | 工程表达 | 验收表达 |
|---|---|---|---|
| 可见性 | 掌握园区关键流量数据 | Probe、Flow、Session、Feature、PCAP | 采集覆盖、吞吐、丢包、解析率 |
| 智能检测 | 识别复杂恶意攻击 | Rule、Behavior、CEP、Campaign、Baseline | 准确率、召回率、误报率、未知攻击测试 |
| 证据闭环 | 告警可解释可回放 | Evidence、PCAP、Graph、Audit | 证据完整率、取证下载、审计留痕 |
| 持续优化 | 误报和新样本能回灌 | Feedback、Whitelist、Rule Review、MLOps | 模型/规则版本、回归报告、发布回滚 |

## 2. 工程系统边界

### 2.1 课题一主责

- 全流量采集：实时流量、离线 PCAP、被动资产发现。
- 多源融合：流量、资产、设备日志、用户行为。
- 实时分析：会话化、特征工程、规则检测、行为检测、CEP。
- 告警运营：告警生成、去重、状态机、指派、反馈、关闭。
- 取证回溯：PCAP 索引、裁剪、下载、证据链。
- 关联分析：实体图谱、攻击链、路径查询。
- 持续学习：反馈驱动白名单、规则复审、模型重训、热更新。
- 验收支撑：样本集、指标公式、压测、第三方测试、试点证明。

### 2.2 非主责但需留接口

- 自动阻断、清洗、防火墙策略下发：由课题二或外部防护设备执行，课题一输出策略建议和证据。
- VPN 专项在网检测：由课题四提供专项模型/规则，课题一提供流量特征、告警融合和降噪上下文。
- 编码增强交换机/智能网关/芯片：由课题三/五承担，课题一提供检测与运营平台接口。

## 4. 核心闭环一：采集到告警

| 步骤 | 输入 | 处理 | 输出 | 失败处理 | 验收证据 |
|---|---|---|---|---|---|
| 1 | 镜像流量/TAP/PCAP | Probe 抓包解析、聚合 | FlowEvent | 本地缓冲、采集指标 | Probe 日志、pps/bps、丢包 |
| 2 | FlowEvent | Ingest 鉴权、限流、去重 | `flow.events.v1` | `dlq.v1` | mTLS 测试、Kafka 消费 |
| 3 | FlowEvent | Session Job 聚合 | SessionEvent/CH sessions | checkpoint 恢复 | Flink RUNNING、session 查询 |
| 4 | SessionEvent | Feature Job 提取特征 | feature_stat/feature_seq | feature DLQ | 特征表、DLQ 计数 |
| 5 | Feature | Rule/Behavior/CEP | Detection/Campaign | 模型/规则回滚 | detection 表、模型版本 |
| 6 | Detection | Alert Generator/Alert Service | Alert/Evidence | 去重、重试 | 告警列表、详情、证据 |

关键设计：每条告警必须能追溯到 `tenant_id`、`event_id`、`community_id`、`session_id`、`feature_set_id`、`rule_version`、`model_version`、`evidence_id`。

## 5. 核心闭环二：告警到取证

告警详情页不是信息展示页，而是取证工作台：

1. 告警摘要：严重级别、置信度、攻击类型、资产、时间窗。
2. 证据摘要：检测原因、规则/模型版本、相关 session、原始流量索引。
3. PCAP 操作：按时间窗、五元组、session_id 发起裁剪。
4. 图谱操作：以源 IP、目标 IP、资产、用户、Campaign 为起点探索。
5. 研判动作：指派、标记 TP/FP、关闭、生成白名单草稿、触发规则复审。
6. 审计留痕：谁在何时看过证据、下载过 PCAP、改过状态。

验收重点：

- completed PCAP 可下载，文件 magic 和 packet 数可验证。
- 证据中的 session/community 与 CH 查询结果一致。
- 图谱探索返回节点和边，并受层级限制保护。
- PCAP 下载必须有权限、有效期和审计。

## 6. 核心闭环三：反馈到优化

| 反馈类型 | 产品动作 | 技术动作 | 验收证据 |
|---|---|---|---|
| TP | 关闭、归档、进入样本库 | 写 AlertFeedback、更新指标 | 反馈记录、样本 ID |
| FP | 选择原因码、生成白名单草稿 | 写 `alert.feedback.v1`，触发规则复审 | 白名单审批记录 |
| 规则问题 | 提交规则复审 | Rule Manager 生成新版本，发 `rule.updates` | 规则版本历史 |
| 模型问题 | 累积反馈样本 | MLOps preflight/train/evaluate/register/activate | 模型评估报告 |
| 数据质量问题 | 标记采集/解析异常 | DLQ 修复重放、数据质量看板 | DLQ 重放报告 |

门禁：模型激活前必须满足样本量、评估指标、回归集、灰度策略和回滚条件。规则发布前必须说明影响范围、测试结果、审批人和回滚版本。

## 7. 数据质量设计

数据质量是检测质量的前置条件。平台应至少展示：

- 采集健康：探针在线、端口、吞吐、丢包、心跳延迟。
- 接入健康：Ingest QPS、拒绝数、限流数、DLQ 数。
- Kafka 健康：topic ISR、consumer lag、写入速率。
- Flink 健康：checkpoint、backpressure、restart、DLQ。
- 存储健康：CH 写入延迟、Merge、Keeper、PG 复制延迟、MinIO 对象错误。
- 语义质量：解析成功率、迟到事件、字段缺失率、特征缺失率。

数据质量异常必须能回答三件事：

1. 影响哪些数据源和时间窗。
2. 是否影响告警可信度。
3. 是否可以通过重放修复。

## 8. 检测质量设计

任务书指标不能只靠工程链路证明，需要算法和测试闭环。

### 8.1 样本体系

| 样本类型 | 用途 | 说明 |
|---|---|---|
| 正常业务样本 | 误报率评估 | 高校/科研/产业/政企园区常见业务 |
| 已知攻击样本 | 准确率/召回率 | 扫描、爆破、C2、横向移动、外传、BOT |
| 未知攻击样本 | 泛化评估 | 留类、时间切分、变种攻击 |
| 加密/隧道样本 | 加密流量分析 | TLS/QUIC/SSH/VPN/隧道 |
| 回放样本 | 回归测试 | 每次规则/模型发布必跑 |

### 8.2 指标公式

```text
Accuracy = (TP + TN) / (TP + FP + TN + FN)
Precision = TP / (TP + FP)
Recall = TP / (TP + FN)
FPR = FP / (FP + TN)
FNR = FN / (FN + TP)
误报率 = FP / (TP + FP) 或按第三方测试约定声明
```

所有报告必须说明采用哪一种误报率口径，避免“正常样本误报率”和“告警误报占比”混用。

## 9. 性能与容量设计

### 9.1 10 x 100Gbps / 512Mpps

必须明确：

- 网卡数量、速率、驱动、队列数、NUMA 绑定。
- 包长分布：64B、小包混合、IMIX、大包。
- 流量类型：正常业务、攻击混合、加密流量、突发流。
- 持续时间：预热、稳定、峰值、恢复。
- 判定指标：丢包率、解析成功率、CPU、内存、队列积压、Kafka lag、Flink backpressure。

### 9.2 P95 <= 60s

端到端延迟必须拆点：

```text
packet_ts
flow_event_ts
ingest_ts
kafka_append_ts
flink_session_out_ts
flink_detection_out_ts
alert_created_ts
api_response_ts
ui_seen_ts
```

报告必须给 P50/P90/P95/P99 和异常样本解释。

## 10. 安全与合规设计

| 域 | 已有基础 | 补强要求 |
|---|---|---|
| 身份认证 | JWT、OIDC、API Token | MFA/Token 生命周期/撤销审计 |
| 接入安全 | Probe-Ingest mTLS | 服务间 mTLS、证书轮换 |
| 总线安全 | Kafka 内网通信 | SASL/SCRAM、TLS、ACL |
| 租户隔离 | tenant_id | Topic、表、缓存、对象路径隔离检查 |
| 权限控制 | RBAC 基础 | 角色-资源-动作矩阵 |
| 取证安全 | 下载链接 | hash、签名、有效期、下载审计 |
| 密钥管理 | K8s Secret | ExternalSecret、轮换、默认值清理 |

## 12. 当前必须如实披露的缺口

| 缺口 | 为什么必须披露 | 下一步 |
|---|---|---|
| 10 x 100Gbps/512Mpps 未专项验证 | 这是任务书/技术设计高指标，不可用局部单测替代 | 制定压测方案并执行 |
| P95 <= 60s 未形成分布报告 | 没有统一时间戳口径 | 增加采样点和 SQL/PromQL |
| 95%/5% 未第三方验证 | 只有工程链路和小样本模型结果 | 建样本集和第三方测试 |
| 模型泛化质量不足 | 小样本 F1=0.8 只是链路证明 | 建立大样本和回归集 |
| Kafka TLS/SASL 未启用 | 生产安全红线 | 做安全加固专项 |
| Secret/ExternalSecret 未完全收口 | 密钥治理风险 | 清理开发默认配置 |
| HA 故障演练不足 | 组件 Running 不等于恢复能力 | 做故障演练报告 |

## 13. 竞争力落地语言

面向评审与售前，建议统一使用：

> 本系统以全流量遥测为底座，以多源融合和实时流计算为核心，以告警证据链、PCAP 取证、实体图谱和反馈学习为闭环，面向园区网络形成“发现-研判-取证-优化-验收”的工程系统。它不是只做大屏、日志汇聚或单点规则检测，而是把任务书中的多源融合、智能检测、准确率/误报率、第三方测试和试点转化要求落成可运行、可演示、可复测、可交付的工程体系。
