## 2. 系统主链路

项目主链路是：

```text
Rust Probe
  -> Ingest Gateway
  -> Kafka
  -> Flink Jobs
  -> ClickHouse / PostgreSQL / OpenSearch / NebulaGraph / Redis / MinIO
  -> Go control-plane APIs
  -> Web UI
  -> Feedback / Whitelist / Rule Review / MLOps
  -> rule.updates / model-updates
  -> Flink hot update
```

该主链路承接的产品闭环是：

```text
全流量可见
  -> 多源融合
  -> 智能检测
  -> 告警研判
  -> PCAP / 图谱取证
  -> 反馈学习
  -> 规则 / 模型治理
  -> 第三方可复测证据
```

## 5. 推荐开发顺序

不使用 loop 引擎时，建议 Codex 目标模式按以下顺序推进：

1. 先做只读基线：`agent.md`、`git status --short`、主设计、验收、状态文档、相关子系统入口。
2. 选择一个 P0 小闭环，不跨太多层同时改。例如先做 `routeManifest + /auth/me + WebSocket 授权时序`，或先做 `DLQ replay 运维 API`。
3. 明确影响面：代码、Proto、Topic、DDL、K8s、Web、测试、文档。
4. 实现时遵循现有模式：Go handler/service/repository，Rust 模块边界，Flink job wiring，Web Ant Design/React Query。
5. 跑最小相关验证：Go/Web/Java/Rust/Proto/MLOps/K8s dry-run，不用无关大测试冒充验收。
6. 做工程复核：状态机、权限、tenant、审计、错误处理、回滚、生产配置、证据层级。
7. 证据落盘：把命令、结果、失败项、未验证边界写进 `doc/02_acceptance/runs/<run_id>/` 或状态文档。
8. 同步状态：只在证据足够时改“已闭环”；否则写“已开发但未验收闭环”或“待专项验证”。
