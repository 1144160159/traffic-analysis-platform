# 模型管理 Flink 真实模型热更新与数据面确认

- 最终 Job ID：`f7b85d363aa31f7d1e8721ae05f6bcc6`

## 最终链路

`Rule Manager 事务状态迁移 -> model_update_outbox -> Kafka model-updates -> Flink broadcast -> MinIO artifact 下载 -> SHA-256 校验 -> XGBoost 加载与 warmup -> 原子切换 tenant+model_id 活跃模型 -> Kafka model-update-applied.v1 -> Rule Manager 聚合 4/4 ack -> action completed + completion audit`

控制面不再把 broker acknowledgement 当成数据面完成。只在所有 Flink 并行子任务均返回 `applied` 后完成关联 rollback action；任一 `failed` ack 会使 action 失败。

## 实现范围

- 模型事件携带 `tenant_id`、稳定 `event_id`、artifact SHA-256 和 threshold。
- Flink 通过配置的 S3/MinIO filesystem 下载 `model.json` 与同目录 `feature_columns.json`。
- artifact 非空且 SHA-256 匹配后，使用 XGBoost4J 2.0.3 真正加载模型并执行 warmup inference。
- 活跃模型按 `tenant_id + model_id` 隔离，并在 warmup 成功后原子切换；旧 native model 延迟 30 秒关闭，避免中断在途推理。
- broadcast state 持久化最近 2048 个处理过的事件，覆盖 A -> B -> replay A 的非连续重放。
- 每个并行子任务发送独立 applied/failed ack；PostgreSQL `model_update_applied_acks` 唯一约束保证幂等聚合。
- TaskManager 运行时镜像增加 XGBoost4J native library 所需的 `libgomp.so.1`，双节点 digest 固定为 `sha256:8c9b090b...771f9`。

## 开发中发现并关闭的真实缺陷

1. r325 只更新版本标记，没有下载、加载和切换模型；严格复审判定 P0。
2. 首次 r326 运行时，operator 中提交端注入的 `ModelRegistry` 因 `transient` 序列化为空；改为在 `open()` 中按 TaskManager runtime 创建，并增加真实 Broadcast harness 测试。
3. artifact 下载和 SHA 校验成功后，XGBoost native library 因基础镜像缺少 `libgomp.so.1` 导致 Job 重启；增加专用 digest-pinned TaskManager 镜像后关闭。
4. Rule Manager 的 broker ack 完成 rollback 行为已移除，改为等待 4/4 data-plane applied ack。
5. r326 后续严格复审发现 4/4 阈值仍来自 ACK 自报的 `parallelism`，且 ACK 制品未绑定 outbox。r328 改为服务端配置/事件快照的可信并行度，显式要求 subtask `COUNT(DISTINCT)=N`、`MIN=0`、`MAX=N-1`，并强制 applied ACK 的 URI 和 SHA-256 与原始事件一致；伪造 1/1、错误 URI/SHA 和缺失指纹均 fail-closed。

## 回退策略

2. 若出现运行时回归，停止当前 Job，使用上一个已知良好 JAR 和对应 savepoint 恢复。
